# 前端界面说明

## 文档目的
这份文档用于帮助后续 agent 或开发者快速理解当前网站前端到底展示了什么、每个页面承担什么职责、页面依赖哪些数据，以及在什么情况下会出现空状态。

本文档基于以下信息整理：
- 前端源码：`frontend/src`
- 页面壳层：`frontend/src/App.js`、`frontend/src/App.css`
- 页面组件：`frontend/src/pages/*`
- 公共组件：`frontend/src/components/*`
- 线上页面实看：`http://115.190.82.67`

说明：
- 本文档重点是“界面理解”，不是 API 设计文档。
- 线上页面的真实展示情况会随数据变化而变化，但页面结构基本稳定。
- 线上实际观察时间：2026-04-06。

## 一眼看懂这个前端

这是一个典型的“后台管理 + 数据分析”型前端，不是营销官网。

整体视觉结构如下：
- 左侧：深色固定导航栏
- 右侧顶部：白色标题栏，显示当前页面标题和“系统在线”状态
- 中间主体：浅灰背景上的卡片、表格、图表、地图

视觉风格特征：
- 以蓝色为主色，辅以绿色、橙色、红色做状态区分
- 大量使用 `Ant Design` 的 `Card`、`Table`、`Statistic`、`Tag`、`Button`
- 分析页和对比页使用 `Recharts`
- 路线和匹配页面使用地图组件
- 数字指标、进度条、状态标签是主视觉重点

## 技术栈与入口

- 前端框架：React 18
- 路由：`react-router-dom` v6
- UI 组件库：Ant Design
- 图表：Recharts
- 请求：Axios
- 地图：默认走百度地图封装，另有 SVG 兜底组件
- 工程类型：CRA（`react-scripts`）

关键入口文件：
- `frontend/src/index.js`
- `frontend/src/App.js`
- `frontend/src/App.css`

接口基地址：
- 前端统一通过 `frontend/src/services/api.js` 请求
- API 基础前缀写死为 `/api`

## 全局框架

全局框架定义在 `frontend/src/App.js`。

### 左侧导航
导航菜单固定包含 7 个入口：
- 仪表板
- 路线管理
- 货物管理
- 匹配结果
- 数据上传
- 数据分析
- 对比分析

导航支持收起：
- 展开时显示图标 + 中文标题
- 收起时只显示图标

Logo 区内容：
- 主标题：运输管理系统
- 副标题：Transport MIS
- 图标文字：TM

### 顶部栏
顶部栏是一个很轻的 header：
- 左边显示当前页面标题
- 右边显示“系统在线”状态徽标

页面标题由当前路由决定：
- `/dashboard` -> 仪表板
- `/routes` -> 路线管理
- `/shipments` -> 货物管理
- `/matching` -> 匹配结果
- `/data-upload` -> 数据上传与算法执行
- `/analytics` -> 数据分析
- `/compare` -> 对比分析

### 内容区
内容区特点：
- 背景为浅灰色
- 纵向滚动
- 页面主体基本都由多个 `Card` 组成
- 常见布局是 KPI 卡 + 表格 + 图表，或 KPI 卡 + 地图 + 详情区

## 5. 数据上传与算法执行

文件：
- `frontend/src/pages/DataUpload/DataUploadPage.jsx`

### 页面定位
这是系统里最偏“操作台”的页面。

它负责三类事情：
- 上传数据
- 启动算法
- 管理场景执行任务

### 页面结构
从上到下依次是：
- 步骤条
- 左右双栏
- 底部场景管理表

### 顶部步骤条
三步流程非常直观：
- 上传数据
- 执行算法
- 对比分析

这说明产品意图很明确：先准备数据，再跑算法，最后比较场景结果。

### 左侧：当前数据文件
有 4 个上传槽位：
- 货物数据 `shipment.csv`
- 路线数据 `route.csv`
- 网络节点 `network.csv`
- 合作参数 `cooperation_parameter.csv`

其中前两个是必填。

这部分界面支持：
- 逐个选择 CSV
- 一键批量上传
- 查看已上传文件状态
- 文件预览
- 删除已上传文件

### 左侧：ZIP 批量场景上传
支持上传一个 ZIP 包：
- 自动扫描压缩包内的场景目录
- 只要目录下有 `route.csv + shipment.csv`，就会注册为独立场景

这说明系统有“多场景实验”能力，而不只是单次上传。

### 右侧：算法执行面板
这里会展示：
- 当前数据是否已满足执行要求
- 执行按钮
- 算法说明
- 运行中耗时
- 运行完成后的简要结果
- 执行失败提示

这个区域属于“提交计算任务”的操作面板。

### 底部：场景管理
表格会展示：
- 场景名称
- 分组
- 路线数
- 货物数
- 状态
- 匹配率
- 操作

并支持：
- 单场景执行
- 批量执行全部场景
- 算法参数弹窗配置

### 算法参数弹窗
弹窗中可调参数包括：
- 最大偏好列表长度
- 最大迭代次数
- 最大不完整稳定匹配次数
- 随机游走概率

这说明前端不只是触发算法，而是允许用户配置算法运行参数。

## 页面之间的业务关系

这几个页面并不是彼此独立的，它们组成的是一条完整的业务链：

1. 数据上传页
- 上传基础数据或批量场景
- 触发算法执行

2. 仪表板
- 快速看哪些场景已经有结果
- 看总体效果和稳定性

3. 路线管理 / 货物管理 / 匹配结果
- 分别从路线、货物、匹配记录三个维度查看明细

4. 数据分析
- 深入看单个场景的结构化分析

5. 对比分析
- 横向比较多个场景结果

如果后续 agent 要理解用户说的“这个网站主要干什么”，最准确的描述是：

这是一个“多场景运输匹配系统”的前端操作台，用于上传运输网络与货物数据、执行稳定匹配算法，并从总览、路线、货物、匹配结果、单场景分析、多场景对比几个维度查看结果。
