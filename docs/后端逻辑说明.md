# 后端逻辑说明

## 文档目的
这份文档用于帮助后续 agent 或开发者快速理解当前项目后端是如何组织的，尤其是：
- Python 后端负责什么
- C++ `stable_match.exe` 负责什么
- 数据文件放在哪里
- “默认数据集”和“场景数据集”两条链路分别怎么跑
- 前端各页面实际依赖哪些后端接口

本文档基于以下内容整理：
- `app.py`
- `app/routes/*`
- `app/services/*`
- `app/models/*`
- `data/`
- `cmake-build-debug/`
- `docker-compose.yml`
- `Dockerfile.backend`

## 一句话理解这个后端

这个后端本质上不是“在线计算平台”，而是一个“Python 包装层 + C++ 算法执行器 + CSV 文件驱动的数据展示平台”。

职责分工可以直接理解为：
- Python/FastAPI：负责接口、文件上传、场景管理、结果解析、聚合分析、前端数据整形
- C++ `stable_match.exe`：负责真正的多式联运稳定匹配算法计算
- CSV 文件：是整个系统的核心输入输出介质

也就是说，Python 并不实现核心匹配算法，它主要负责“把数据喂给 exe，把结果从 CSV 读出来，再转成前端能用的 JSON”。

## 后端整体架构

### 入口
后端入口文件是：
- `app.py`

它负责：
- 创建 FastAPI 应用
- 注册 CORS
- 注册异常处理
- 挂载路由模块

挂载的主要路由：
- `app/routes/data_routes.py`
- `app/routes/query_routes.py`
- `app/routes/matching.py`
- `app/routes/analytics.py`
- `app/routes/scenes.py`

### 分层结构
项目后端大致分成四层：

1. 路由层
- 接收 HTTP 请求
- 做参数校验
- 调用 service
- 返回 JSON

2. 服务层
- 负责数据加载、缓存、结果解析、算法调用

3. 模型层
- 负责把 CSV 行解析成 Python 对象

4. 文件层
- 真正的“数据库”其实是 CSV 文件和 `scenes.json`

## 核心目录和文件角色

### 1. 默认数据目录
- `data/`

这个目录保存默认数据集，主要文件有：
- `shipment.csv`
- `route.csv`
- `network.csv`
- `cooperation_parameter.csv`
- `scenes.json`
- `data/scenes/<scene_id>/...`

可以把它理解成：
- `data/` 根目录：默认数据集
- `data/scenes/`：多场景实验数据集
- `data/scenes.json`：场景注册表

### 2. 算法目录
- `cmake-build-debug/`

关键文件：
- `cmake-build-debug/stable_match.exe`
- `cmake-build-debug/result/stable_matching.csv`

其中：
- `stable_match.exe` 是 C++ 算法可执行文件
- `result/stable_matching.csv` 是默认数据集的结果输出位置

### 3. Python 服务单例
- `app/services/__init__.py`

这里创建了应用级单例：
- `data_loader`
- `data_service`
- `matching_service`

这意味着默认数据集相关接口共享同一份缓存和服务实例。

## 两条核心业务链路

这是理解整个后端最重要的部分。

这个项目实际上有两条并行的业务链路：

1. 默认数据集链路
- 面向 `data/` 根目录中的单组数据
- 用于基础数据展示、单次算法执行

2. 场景数据集链路
- 面向 `data/scenes/<scene_id>/` 中的多组实验数据
- 用于场景管理、批量运行、对比分析、单场景分析

后续 agent 如果不先分清这两条链路，很容易读代码时混乱。

## 一、默认数据集链路

### 适用接口
这一组接口主要依赖 `app/services/__init__.py` 中的单例服务：
- `/api/network`
- `/api/shipments`
- `/api/routes`
- `/api/search/shipments`
- `/api/filter/routes`
- `/api/matchings`
- `/api/matchings/detailed`
- `/api/summary`
- `/api/matching/execute`
- `/api/matching/status`

### 数据来源
默认链路理论上读取：
- `DATA_DIR/shipment.csv`
- `DATA_DIR/route.csv`
- `DATA_DIR/network.csv`
- `DATA_DIR/cooperation_parameter.csv`

其中 `DATA_DIR` 来自：
- `app/config.py`

### 默认链路的数据流

#### 读取路线
调用链如下：

`/api/routes`
-> `app/routes/data_routes.py:get_routes_data`
-> `data_service.get_all_routes()`
-> `data_loader.load_routes()`
-> `Route.from_csv_row()`

#### 读取货物
调用链如下：

`/api/shipments`
-> `app/routes/data_routes.py:get_shipments_data`
-> `data_service.get_all_shipments()`
-> `data_loader.load_shipments()`
-> `Shipment.from_csv_row()`

#### 读取匹配结果
调用链如下：

`/api/matchings`
-> `app/routes/matching.py:get_all_matchings`
-> `matching_service.get_all_matchings()`
-> `data_loader.load_matchings()`
-> `StableMatching.from_csv_rows()`

### 默认算法执行链路
这一条链路由：
- `app/routes/matching.py`
- `app/services/matching_service.py`

共同完成。

调用链如下：

`POST /api/matching/execute`
-> 在线程池中执行 `_run_and_record`
-> `matching_service.execute_matching_algorithm()`
-> `subprocess.Popen(...)`
-> 调用 `stable_match.exe`
-> 等待输出 `cmake-build-debug/result/stable_matching.csv`
-> 重新解析结果并返回 summary

### 默认算法输入输出
`MatchingService.execute_matching_algorithm()` 里传给 exe 的输入是：
- `/app/data/network.csv`
- `/app/data/shipment.csv`
- `/app/data/route.csv`
- `/app/data/cooperation_parameter.csv`
- `/app/cmake-build-debug/result/stable_matching.csv`

也就是说，默认模式下：
- 输入来自默认数据目录
- 输出写到固定的默认结果文件

### 默认任务状态
`app/routes/matching.py` 内维护了一个进程内任务表：
- `_tasks`
- `_CURRENT_TASK_KEY`

特点：
- 不落库
- 不持久化
- 重启进程后任务状态会丢
- 适合“提交任务 -> 前端轮询状态”这种轻量用法

## 二、场景数据集链路

### 适用接口
这一组接口主要由 `app/routes/scenes.py` 和 `app/routes/analytics.py` 负责：
- `/api/scenes`
- `/api/scenes/{scene_id}/run`
- `/api/scenes/{scene_id}/status`
- `/api/scenes/run-all`
- `/api/scenes/{scene_id}/shipments`
- `/api/scenes/{scene_id}/routes`
- `/api/scenes/{scene_id}/matchings`
- `/api/compare`
- `/api/upload/zip`
- `/api/analytics/*`

### 场景注册表
场景元数据保存在：
- `data/scenes.json`

每个场景一般包含：
- `id`
- `label`
- `group`
- `sub`
- `route_count`
- `shipment_count`
- `result_file`
- `status`

当前本地仓库里的 `data/scenes.json` 已经注册了 11 个场景，例如：
- 运价+时间
- 时间价值
- 政府补贴
- 政府补贴/补贴20%
- ...
- 政府补贴/补贴90%

### 场景输入目录
每个场景一个目录：
- `data/scenes/<scene_id>/`

每个场景目录通常包含：
- `shipment.csv`
- `route.csv`
- `network.csv`
- `cooperation_parameter.csv`

### 场景执行链路
调用链如下：

`POST /api/scenes/{scene_id}/run`
-> `app/routes/scenes.py:run_scene`
-> 将任务放入串行队列
-> `_dispatch_next()`
-> `_run_scene(task_id, scene_id, algo_params)`
-> `subprocess.Popen(...)`
-> 调用 `stable_match.exe`
-> 输出到 `cmake-build-debug/result/<scene_id>/stable_matching.csv`
-> 解析结果摘要
-> 更新 `_tasks` 和 `scenes.json`

### 场景执行与默认执行的最大区别
默认执行和场景执行都调同一个 exe，但运行方式不同：

默认执行：
- 面向单组默认数据
- 输出固定文件
- 使用 `matching_service.execute_matching_algorithm()`
- 不带算法参数

场景执行：
- 面向指定 `scene_id`
- 每个场景结果单独存目录
- 在 `scenes.py` 中直接 `subprocess.Popen`
- 支持额外传 4 个算法参数
- 支持串行队列和批量提交

### 场景任务队列
`app/routes/scenes.py` 内实现了一套简单的串行队列：
- `_queue`
- `_queue_running`
- `_pool = ThreadPoolExecutor(max_workers=1)`

特点：
- 同一时刻只跑一个场景算法进程
- 后续任务排队
- 前端可以看到 `queued / running / done / failed`

这么设计的目的很明显：
- 避免同时启动多个 `stable_match.exe` 把服务器压垮

### 批量运行全部场景
`POST /api/scenes/run-all`

作用：
- 把所有未运行中的场景加入串行队列
- 逐个执行

这个接口是“对比分析”功能的重要前置入口，因为只有场景都跑完，对比页才有横向数据。

## C++ 算法 exe 的角色

### 可执行文件
- `cmake-build-debug/stable_match.exe`

这个文件是整个系统的真正核心计算引擎。

Python 只做两件事：
- 组织输入参数
- 读取输出结果

### 输入参数
从代码看，exe 至少支持如下输入：
- `network.csv`
- `shipment.csv`
- `route.csv`
- `cooperation_parameter.csv`
- 输出结果文件路径

在场景模式下，还会额外追加：
- `max_prefer_list`
- `max_iter`
- `max_incomplete`
- `prob_random_walk`

也就是说，这个 exe 不只是“黑盒子运行”，它还支持通过命令行传递算法参数。

### 输出结果
输出 CSV 叫：
- `stable_matching.csv`

这个文件是整个系统的“结果真相来源”。

前端看到的大多数匹配结果、汇总指标、分析图表，本质上都来自对这个 CSV 的二次解析。

---

## `stable_matching.csv` 的格式

这个结果文件格式在：
- `app/models/matching.py`
- `app/routes/scenes.py:_parse_result`
- `app/routes/analytics.py:_parse_matchings`

都有解析逻辑。

结构大致如下：

第 1 行：
- `Shipment,1,2,3,...`
- 表示货物 ID 列表

第 2 行：
- `Route,10,10,4,...,Self,...`
- 表示每个货物对应分配到的路线
- `Self` 表示未匹配

第 3~9 行：
- 总容量
- 货物总集装箱数
- 已匹配集装箱数
- 是否稳定
- 迭代次数
- 重启次数
- CPU 时间

本地仓库里 `cmake-build-debug/result/stable_matching.csv` 的内容表明：
- 这是一个“单次默认数据集”的结果文件
- 当前本地只看到默认结果文件，没有看到 `result/<scene_id>/stable_matching.csv`

## CSV 输入文件的语义

### 1. `shipment.csv`
由 `Shipment.from_csv_row()` 解析。

核心字段：
- `shipment index`
- `Origin`
- `destination`
- `Demand`
- `Time value(CNY/TEU)`

模型语义：
- 一个货物就是一条运输需求
- `Demand` 以 TEU 为单位
- `weight` 和 `volume` 不是直接存储，而是由 TEU 估算出来

### 2. `route.csv`
由 `Route.from_csv_row()` 解析。

结构特点：
- 第一行是元数据
- 第二行是标题
- 后面每行是一路线

模型语义：
- 一条路线包含节点序列
- 有每段运费、每段时间、总容量
- `route_category` 并不是从 CSV 直接读，而是根据 `route_id` 区间推断：
  - 1~9：西海路新通道
  - 10~12：长江经济带
  - 13~17：跨境公路

### 3. `network.csv`
当前 Python 端几乎不做复杂解析。

原因是：
- 实际前端展示用的城市名和经纬度不是从 `network.csv` 动态解析，而是写死在 `DataService.city_coordinates` 里

这意味着：
- `network.csv` 更像是 exe 算法需要的输入
- 前端地图所需城市信息主要由 Python 代码内置

### 4. `cooperation_parameter.csv`
Python 端几乎不直接分析内容，主要作为 exe 输入文件。

## 模型层的职责

### `Shipment`
文件：
- `app/models/shipment.py`

职责：
- 把 `shipment.csv` 的行解析成货物对象
- 提供 `weight`、`volume` 这种派生字段

### `Route`
文件：
- `app/models/route.py`

职责：
- 把 `route.csv` 的行解析成路线对象
- 计算总成本、总耗时、可用容量、利用率等属性

### `StableMatching`
文件：
- `app/models/matching.py`

职责：
- 把 `stable_matching.csv` 解析成结构化匹配结果
- 提供：
  - 总货物数
  - 已匹配数
  - 未匹配数
  - 匹配率
  - 集装箱匹配率
  - 路线使用统计

## 服务层的职责

## 1. `DataLoader`
文件：
- `app/services/data_loader.py`

职责：
- 纯粹负责“从 CSV 文件加载对象”
- 不太关心前端需要什么字段

它是偏底层的 I/O 解析器。

## 2. `DataService`
文件：
- `app/services/data_service.py`

职责：
- 基于 `DataLoader` 的原始对象，拼出前端更方便使用的数据结构
- 补充城市名、经纬度、总距离等展示字段
- 提供搜索与筛选
- 做内存缓存

这个类是典型的“面向前端展示”的整形层。

### `DataService` 的重要特点

#### 城市坐标是硬编码的
`DataService.city_coordinates` 里写死了 9 个城市：
- 成都
- 重庆
- 贵阳
- 怀化
- 北部湾
- 上海
- 胡志明
- 曼谷
- 新加坡

所以前端地图和城市名展示高度依赖这份内置映射。

#### 路线距离是 Python 现场算的
`get_all_routes()` 会使用 Haversine 公式根据硬编码经纬度计算总距离。

也就是说：
- `route.csv` 不直接提供“公里数”
- 前端显示的总距离是 Python 补算的

## 3. `MatchingService`
文件：
- `app/services/matching_service.py`

职责：
- 读取默认结果文件
- 汇总匹配摘要
- 计算路线利用率
- 调用默认模式下的 `stable_match.exe`

这个类是默认数据集链路的核心服务。

## 分析接口是怎么来的

文件：
- `app/routes/analytics.py`

这些接口并不是直接从数据库查来的，而是“现算聚合结果”。

关键设计是：
- 如果传 `scene_id`，就临时创建该场景的 `DataLoader + DataService`
- 再从该场景的 `stable_matching.csv` 重建 matchings
- 然后做聚合分析

这说明分析页的数据来源是：
- 场景原始 CSV
- 场景结果 CSV

而不是单独存储过的分析结果。

### 分析接口包括
- `/api/analytics/route-utilization`
- `/api/analytics/od-flow`
- `/api/analytics/time-value`
- `/api/analytics/algorithm-quality`

它们分别负责：
- 路线利用率图
- OD Sankey 图
- 时间价值分档图
- 算法质量面板

## 对比接口是怎么来的

文件：
- `app/routes/scenes.py`

核心接口：
- `/api/compare`

它的逻辑不是直接读取一个“对比结果表”，而是：
- 遍历 `scenes.json` 中的每个场景
- 找到场景结果
- 解析每个场景的匹配率、稳定性、CPU 时间等指标
- 再附加平均运费和路线分流量
- 最后返回前端对比页所需的数据

所以“对比分析”本质上是一个动态聚合接口。

## 文件上传逻辑

文件：
- `app/routes/data_routes.py`
- `app/routes/scenes.py`

### 单文件/批量 CSV 上传
`data_routes.py` 负责默认数据集上传：
- 单文件上传 `/api/upload`
- 批量上传 `/api/upload/batch`
- 上传历史 `/api/uploads`
- 文件预览 `/api/uploads/preview/{filename}`
- 文件删除 `/api/uploads/{filename}`

特点：
- 上传的文件会直接覆盖默认数据目录中的同名文件
- 上传时间记录在 `.upload_records.json`
- 上传后会清 `data_service` 缓存

### ZIP 上传
`scenes.py` 负责场景化 ZIP 上传：
- `/api/upload/zip`

它会：
- 解压 zip
- 扫描其中哪些目录同时包含 `shipment.csv + route.csv`
- 把这些目录注册为新的场景
- 写入 `data/scenes.json`
- 把文件落到 `data/scenes/<scene_id>/`

这就是“批量实验场景导入”能力的来源。

## 部署视角下的路径关系

### Docker 中的路径
从 `Dockerfile.backend` 和 `docker-compose.yml` 看：

容器工作目录是：
- `/app`

关键路径：
- `/app/data`：默认数据目录
- `/app/cmake-build-debug/stable_match.exe`：算法 exe
- `/app/cmake-build-debug/result`：算法结果目录

volume 挂载：
- `tms_data` -> `/app/data`
- `tms_result` -> `/app/cmake-build-debug/result`

镜像初始化时：
- `data/` 会先被复制到 `/app/data_init/`
- 启动时 `docker-entrypoint.sh` 再把初始化数据同步到 `/app/data`

这说明生产环境里：
- 数据和结果都会持久化
- 镜像里的初始数据只在 volume 空时补充

## 当前代码里最容易让 agent 混淆的点

### 1. 默认链路和场景链路不是同一套实现
不要以为 `/api/matching/execute` 和 `/api/scenes/{id}/run` 只是参数不同。

实际上：
- 它们分别走不同代码路径
- 一个走 `MatchingService`
- 一个在 `scenes.py` 里直接调 subprocess

### 2. `stable_matching.csv` 有“默认结果文件”和“场景结果文件”两种位置
默认：
- `cmake-build-debug/result/stable_matching.csv`

场景：
- `cmake-build-debug/result/<scene_id>/stable_matching.csv`

### 3. 本地开发和 Docker 部署下的数据目录取法不完全一致
这是当前代码里很重要的一个现实细节。

`app/config.py` 的默认 `DATA_DIR` 是：
- `./cmake-build-debug/data`

但：
- `app/routes/scenes.py` 使用的是项目根目录下的 `data/`
- `app/routes/analytics.py` 也使用项目根目录下的 `data/`
- Docker 环境又通过环境变量把默认数据目录设成 `/app/data`

这意味着：
- 在 Docker/生产环境里，各模块大体是统一的
- 在本地裸跑时，默认数据链路和场景链路可能读取的不是同一套目录

后续 agent 如果调试本地接口，一定要先确认当前模块到底读的是哪一个数据目录。

### 4. 默认算法执行路径强依赖容器内 `/app/...`
`MatchingService.execute_matching_algorithm()` 里硬编码了：
- `/app/data/...`
- `/app/cmake-build-debug/...`

这说明默认算法执行链路主要是按 Docker 部署环境写的。

而 `scenes.py` 的场景执行链路使用的是项目根相对路径，更适合本地仓库结构。

### 5. 本地仓库当前只有默认结果文件
从当前仓库内容看：
- 有 `cmake-build-debug/result/stable_matching.csv`
- 但没有看到 `cmake-build-debug/result/<scene_id>/stable_matching.csv`

这说明当前本地仓库里：
- 默认数据结果是现成的
- 场景结果文件大概率尚未在本地生成或未纳入仓库

这能解释为什么有些页面结构已经写好，但并不一定在本地都有完整数据。

如果要最短时间理解后端，建议按这个顺序读：

## 一句话总结

这个后端的本质不是“Python 算法服务”，而是一个围绕 CSV 文件和 `stable_match.exe` 搭起来的多场景实验平台：Python 管理数据和接口，C++ 执行稳定匹配，前端只是在展示这些文件驱动出来的结果。
