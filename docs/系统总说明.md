## 一句话理解整个项目

这是一个围绕“多式联运稳定匹配算法”搭建的实验与展示平台：
- 前端负责展示页面、上传数据、触发算法、查看分析结果
- FastAPI 后端负责文件管理、场景管理、结果解析和接口输出
- C++ `stable_match.exe` 负责真正的匹配算法计算
- CSV 文件是系统的主要输入输出介质

换句话说，这个项目本质上是一个“算法展示平台”，不是一个传统数据库驱动的业务系统。

## 系统的核心闭环

这个系统的主业务闭环非常清晰：

1. 准备数据
- 用户上传默认数据集，或者批量导入多个场景数据集

2. 执行算法
- 后端调用 C++ `stable_match.exe`
- 生成 `stable_matching.csv`

3. 解析结果
- Python 把结果 CSV 解析成结构化 JSON

4. 前端展示
- 仪表板看总览
- 路线/货物/匹配结果看明细
- 数据分析看单场景
- 对比分析看多场景

所以这个系统不是“用户实时输入条件然后即时求解”的纯在线求解器，而更像是“离线执行 + 在线展示”的实验平台。

## 前端在系统里的位置

从系统角度看，前端不是算法主体，而是一个“控制台 + 结果看板”。

### 前端承担的角色
- 选择页面视图
- 上传 CSV 或 ZIP
- 提交算法任务
- 查询任务状态
- 查看路线、货物、匹配结果
- 查看单场景分析和多场景对比

### 前端不承担的角色
- 不做算法计算
- 不持久化复杂业务数据
- 不自己推导结果，只消费后端已经聚合好的 JSON

### 页面功能分工
- 仪表板：看整体场景结果概览
- 路线管理：看路线地图、路线详情和路线表格
- 货物管理：看货物数据
- 匹配结果：看匹配详情、利用率图和匹配表
- 数据上传：上传数据、执行算法、管理场景任务
- 数据分析：单场景深入分析
- 对比分析：多场景横向比较

## 后端在系统里的位置

从系统角度看，后端是“协调者”和“转换器”。

它不直接实现匹配算法，但它把整个系统真正连接了起来。

### 后端承担的角色
- 暴露给前端的 API
- 把上传的文件落地到数据目录
- 管理 `scenes.json`
- 启动 `stable_match.exe`
- 读取 `stable_matching.csv`
- 把 CSV 转成前端友好的 JSON
- 做分析图表需要的聚合计算

### 后端不承担的角色
- 不自己做核心匹配求解
- 不依赖关系型数据库作为核心状态来源

## 算法 exe 在系统里的位置

### 为什么它是核心
整个项目真正“有技术壁垒”的部分，其实就在：
- `cmake-build-debug/stable_match.exe`

因为：
- 匹配率怎么来
- 哪个货物分到哪条路线
- 是否稳定
- 迭代次数是多少
- CPU 时间是多少

这些都不是 Python 算出来的，而是 exe 算出来的。

### Python 与 exe 的关系
Python 做的是：
- 准备输入路径
- 传参数
- 启动进程
- 等待结果文件
- 解析结果文件

所以 Python 更像“算法外壳”和“接口层”。

## 数据文件在系统里的位置

这个系统最重要的一点是：它几乎是完全 CSV 驱动的。

### 输入文件
主要有 4 类：
- `shipment.csv`：货物需求
- `route.csv`：路线与容量
- `network.csv`：网络节点
- `cooperation_parameter.csv`：合作参数

### 输出文件
核心就是：
- `stable_matching.csv`

### 场景机制
场景机制的本质不是数据库表，而是：
- 为每个实验准备一套独立输入 CSV
- 为每个实验生成一份独立结果 CSV
- 再用 `scenes.json` 做统一登记

所以可以把 `scenes.json` 理解为：
- 轻量的场景注册表
- 而不是完整业务数据库

## 两条运行模式

这是整个系统必须分清的一个概念。

### 模式一：默认数据模式
面向：
- `data/` 根目录中的默认数据

特点：
- 用于基础数据展示
- 用于单次算法执行
- 输出固定结果文件

常见接口：
- `/api/shipments`
- `/api/routes`
- `/api/matchings`
- `/api/matching/execute`

### 模式二：场景模式
面向：
- `data/scenes/<scene_id>/` 中的实验场景

特点：
- 每个场景有独立输入文件
- 每个场景有独立结果文件
- 支持排队执行
- 支持横向对比

常见接口：
- `/api/scenes`
- `/api/scenes/{scene_id}/run`
- `/api/scenes/{scene_id}/shipments`
- `/api/scenes/{scene_id}/routes`
- `/api/scenes/{scene_id}/matchings`
- `/api/compare`
- `/api/analytics/*`

## 一次典型请求是怎么流动的

下面用几个最常见的前端页面举例。

### 例子 1：打开仪表板
前端会请求：
- `/api/compare`
- `/api/scenes`

后端会：
- 从 `scenes.json` 读取所有场景
- 查每个场景是否有结果
- 解析结果摘要
- 返回场景卡片需要的数据

### 例子 2：在数据上传页点击“执行算法”
前端会请求：
- `POST /api/matching/execute`

后端会：
- 起一个后台线程
- 调用 `stable_match.exe`
- 等待默认结果文件生成
- 前端轮询 `/api/matching/status`

### 例子 3：执行某个场景
前端会请求：
- `POST /api/scenes/{scene_id}/run`

后端会：
- 把任务放进场景队列
- 串行启动 exe
- 输出到该场景的结果目录
- 前端轮询 `/api/scenes/{scene_id}/status`

### 例子 4：打开对比分析页
前端会请求：
- `/api/compare`

后端会：
- 遍历场景
- 解析每个场景的结果 CSV
- 计算匹配率、集装箱率、CPU 时间、平均运费、路线分流量
- 返回用于对比图和表格的数据

## 当前系统最重要的实现特点

### 1. 没有重量级数据库依赖
项目的核心数据不是存在 MySQL / Postgres 里，而是存在 CSV 和 `scenes.json` 里。

### 2. 后端偏“文件驱动”
很多接口本质上不是查数据库，而是：
- 读 CSV
- 解析 CSV
- 聚合统计
- 返回 JSON

### 3. 分析页和对比页是“动态聚合”
它们不是读取预计算分析表，而是根据原始 CSV 和结果 CSV 现场计算出图表数据。

### 4. 场景任务是进程内队列
没有用 Celery、Redis 队列这类重型方案，而是 Python 进程内维护的轻量任务状态与串行执行队列。

### 5. 路线和城市展示有一层“前端友好整形”
例如：
- 城市名和经纬度由 Python 补充
- 路线总距离由 Python 用经纬度计算
- 路线分类由 `route_id` 区间推断

## 部署形态

### Docker 形态
生产环境主要按 Docker 思路部署。

#### 后端容器
职责：
- 运行 FastAPI
- 持有 `stable_match.exe`
- 持有默认数据目录
- 负责执行算法

#### Nginx 容器
职责：
- 托管前端打包结果
- 反向代理 API

### 关键挂载
- `/app/data`：持久化上传后的默认数据和场景数据
- `/app/cmake-build-debug/result`：持久化算法结果

### 初始化逻辑
镜像构建时会带入一份初始 `data/`，容器启动时：
- 如果 volume 里没有对应文件，就从镜像初始化进去
- 已有文件不会被覆盖

这意味着生产环境可以保留用户上传后的数据状态。

## 当前仓库下的已知现实情况

结合当前仓库和线上站点，可以得到这些结论：

### 1. 前端骨架已经完整
前端页面结构、卡片布局、图表容器、地图页面都已经做完。

### 2. 后端逻辑已经覆盖主要闭环
上传、执行、场景、分析、对比都已有接口。

### 3. 算法是外部 exe，不在 Python 里实现
这点必须牢牢记住，否则会误以为 `matching_service.py` 里有核心求解逻辑。

### 4. 部分页面当前是否有数据，取决于结果文件是否存在
比如：
- 仪表板和路线管理更容易看到现成数据
- 货物、匹配、分析、对比有时会因为对应数据或结果未准备好而出现空状态

### 5. 本地路径和容器路径有差异
这是当前代码里一个非常值得注意的实现细节，调试时一定要确认当前模块究竟在读哪套目录。

## 推荐的系统认知方式

如果后续有人问“这个项目到底是什么”，最好的回答方式不是从技术栈开始，而是从系统角色开始：

1. 这是一个多式联运稳定匹配实验平台
2. C++ exe 负责核心求解
3. Python 负责把文件、场景、结果和前端连接起来
4. 前端负责把这些结果做成可操作、可对比、可分析的页面

## 一句话总结

整个项目可以理解成：

一个以 `stable_match.exe` 为算法核心、以 CSV 为数据载体、以 FastAPI 为组织层、以 React 为展示层的多场景多式联运匹配分析平台。
