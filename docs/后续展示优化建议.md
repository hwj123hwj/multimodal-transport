# 后续展示优化建议

## 文档目的
这份文档不是让现在立刻大改代码，而是给后续迭代提供方向。

适用场景：
- 后面补充了更多业务知识，希望页面展示更“像业务系统”而不只是“算法展示台”
- 需要明确哪些地方值得继续增强，哪些地方暂时不用动
- 需要提醒后续 agent 目前哪些技术细节会影响展示效果

一句话定位：

当前系统已经具备“展示算法结果”的完整框架，后续更值得做的是增强“业务解释能力”和“决策支持能力”，而不是重做架构。

## 当前系统已经做对的事情

这部分是建议保留的，不需要推翻。

### 1. 页面框架已经完整
目前前端已经形成了很清晰的产品闭环：
- 数据上传
- 算法执行
- 总览
- 明细查看
- 单场景分析
- 多场景对比

这个结构非常适合继续演进，不需要重做信息架构。

### 2. 路线页和仪表板已经具备展示核心
当前最成熟的页面是：
- 仪表板
- 路线管理

这两个页面已经能承担“结果展示”和“路线说明”的基本职责，后续只需要补充业务语义，不需要推翻页面结构。

### 3. 场景机制很适合做实验平台
`data/scenes/ + scenes.json + compare` 这一整套机制天然适合：
- 展示政策场景
- 展示参数敏感性分析
- 展示不同业务假设下的结果变化

这其实是项目很强的一个特点，后续应该继续围绕它做“业务化表达”。

## 当前系统更像什么

目前这个系统更像：
- 算法实验结果展示平台

但如果以后补充更多业务知识，它可以逐步往下面这个方向靠：
- 面向运营/规划/政策评估的多式联运决策辅助平台

也就是说，后续优化重点不应只是“再多几个图表”，而应该是：
- 把算法指标翻译成业务语言
- 把场景差异翻译成业务影响
- 把结果展示升级成“为什么值得看”

## 后续最值得加强的 4 个方向

## 1. 把算法指标翻译成业务指标

当前页面里大量展示的是：
- 匹配率
- 集装箱率
- CPU 时间
- 迭代次数
- 稳定性

这些指标对研发有价值，但对业务角色来说还不够直观。

后续更建议补充或强调这些表达：
- 平均运输成本变化
- 高价值货物保障情况
- 各路线类别的容量利用情况
- 未匹配货物占比及原因
- 补贴政策带来的改善幅度
- 不同场景下的成本-效率平衡

建议原则：
- 保留算法指标
- 但让算法指标退居二线
- 把业务结果放到更前面

## 2. 把“场景”讲成“业务故事”

现在场景名称已经有一定业务语义，比如：
- 运价+时间
- 时间价值
- 政府补贴/补贴20%

但页面里还没有把这些场景讲清楚。

后续更建议让每个场景都具备这三类信息：
- 场景假设是什么
- 这个场景为什么重要
- 这个场景相对于基线场景改变了什么

最适合补充的位置：
- 仪表板场景卡片
- 数据分析页顶部
- 对比分析页的场景选择区

最简单可行的做法不是改结构，而是先补：
- 场景说明文字
- Tooltip
- 标签
- “基线场景 / 政策场景 / 参数场景”分类

## 3. 把“结果”升级成“解释”

现在系统能告诉用户“结果是什么”，但还不太能告诉用户“为什么会这样”。

以后最有价值的增强方向包括：
- 为什么某些货物未匹配
- 为什么某条路线利用率高
- 为什么政府补贴提高了匹配率
- 为什么高时间价值货物更容易获得匹配

这类解释不一定要一开始就做复杂智能分析。

可以先用规则型解释：
- 未匹配可能因为容量不足
- 未匹配可能因为路径覆盖不足
- 某场景改善主要来自某一类路线利用率提升
- 某场景改善主要来自高价值货物流向变化

这会明显提升页面的“业务说服力”。

## 4. 把对比页做成“决策页”

当前对比分析页更像“实验结果面板”。

后续更建议它承担“结论页”的角色。

可以逐步强化这些能力：
- 自动标出最优场景
- 自动标出最稳妥场景
- 自动标出成本最低场景
- 自动标出高价值货物保障最优场景
- 给出“适合什么业务目标”的结论标签

例如：
- 如果目标是最高匹配率，推荐场景 A
- 如果目标是控制运费，推荐场景 B
- 如果目标是兼顾稳定性与效果，推荐场景 C

这会让对比页从“看图”升级成“辅助判断”。

## 页面级优化建议

## 1. 仪表板

当前优势：
- 已经是系统最直观的总览页
- 场景卡片形式很适合继续扩展

后续建议优先补充：
- 场景类别标签：基线/政策/参数
- 场景说明摘要：一句话说明该场景在模拟什么
- 关键业务 KPI：例如平均成本、高价值货物保障、未匹配比例
- “推荐查看”入口：从仪表板直接跳到最值得关注的场景分析页

不建议现在大改：
- 卡片网格布局
- KPI 区结构

## 2. 路线管理

当前优势：
- 地图 + 详情 + 表格三段式结构已经很完整
- 适合承载业务解释

后续建议优先补充：
- 路线类别说明：西海路新通道、长江经济带、跨境公路分别代表什么
- 路线业务属性：适合高时效/低成本/跨境/干线等
- 关键路段瓶颈提示
- 利用率说明，不只是显示数值

最值得增加的业务表达：
- 为什么这条路线重要
- 它服务了哪些目的地
- 它在场景切换后利用率变化如何

## 3. 货物管理

当前状态：
- 页面结构完整，但线上经常处于空状态

后续建议优先补充：
- 货物分层：普通货物、高价值货物、重点保障货物
- OD 分组视角
- 时间价值解释
- 未匹配货物筛选

如果以后要增强业务味道，这一页很适合做：
- “需求侧画像页”

## 4. 匹配结果

当前优势：
- 已有地图、详情、表格、利用率图

后续建议优先补充：
- 未匹配原因解释
- 匹配方案解释
- 货物与路线的适配理由
- 某条路线承载了多少高价值货物

这一页未来最适合强化成：
- “结果解释页”

## 5. 数据上传与算法执行

当前优势：
- 已经具备完整的操作台能力

后续建议优先补充：
- 数据模板说明
- 场景命名规范说明
- 算法参数业务含义解释
- 上传后会影响哪些页面的提示

这一页不必追求更花哨，而应追求：
- 低误操作
- 强说明性

## 6. 数据分析

当前优势：
- 图表类型已经足够多
- 路线利用率、时间价值、OD、算法质量四块结构合理

后续建议优先补充：
- 结论摘要区：这个场景说明了什么
- 与基线场景的差异
- 高价值货物保障情况
- 成本变化方向
- 哪些路线从中受益最大

这一页未来最适合强化成：
- “单场景业务复盘页”

## 7. 对比分析

当前优势：
- 已具备多图表、多指标、多场景对比能力

后续建议优先补充：
- 基线场景固定标识
- 政策场景分组展示
- 自动结论摘要
- 业务目标筛选器

这一页未来最适合强化成：
- “场景决策页”

## 未来最值得补充的业务知识类型

如果后面你会继续补业务知识，优先补这些最有价值：

### 1. 场景定义
每个场景到底模拟什么业务假设。

例如：
- 运价变化
- 时间价值变化
- 政府补贴变化
- 容量变化
- 路线开放/关闭

### 2. 路线业务意义
每一类路线在实际业务中代表什么。

例如：
- 哪类偏低成本
- 哪类偏高时效
- 哪类偏跨境
- 哪类更容易受政策影响

### 3. 货物分层逻辑
不同时间价值、不同 OD 的货物在业务上意味着什么。

### 4. 决策目标
系统最终是服务什么目标：
- 提高匹配率
- 控制成本
- 提高稳定性
- 提高高价值货物保障能力
- 验证政策有效性

只要这几类业务知识补进来，页面就能明显从“技术展示”升级成“业务展示”。

## 当前代码层面最值得记住的技术注意点

这些不一定是马上要修的 bug，但后续展示增强时一定会遇到。

### 1. 默认数据链路和场景链路是分开的
后续如果某个页面要同时支持默认数据和场景数据，一定要先确认它当前走的是哪条链路。

### 2. 结果依赖 `stable_matching.csv`
很多页面之所以空，不是因为前端坏了，而是因为对应结果文件不存在或没被读取到。

### 3. 路线和城市信息有一部分是 Python 补出来的
尤其是：
- 城市名称
- 经纬度
- 距离
- 路线分类

所以如果后面要扩展地图和路线语义，要先确认这些字段是来自 CSV 还是来自 Python 代码内置逻辑。

### 4. 本地路径和 Docker 路径不完全统一
后续如果有人调试“为什么某页没数据”，第一件事应该先确认当前代码到底在读哪个数据目录。

### 5. 场景分析和对比是“动态聚合”不是“持久化报表”
所以如果后续要增加更多图表，不一定需要新存储表，很多情况下可以继续沿用现算聚合的方式。

## 推荐的后续迭代顺序

如果以后真的要动展示，我建议按这个顺序来，性价比最高：

### 第一阶段：补文案和语义
- 场景说明
- 路线类别说明
- 指标业务解释
- Tooltip 和标签

这个阶段几乎不动架构，收益最大。

### 第二阶段：补结论摘要
- 单场景结论摘要
- 多场景推荐结论
- 未匹配原因摘要

这个阶段能明显提升“业务可读性”。

### 第三阶段：补业务视角筛选
- 按业务目标筛选场景
- 按货物层级查看结果
- 按路线类别查看变化

### 第四阶段：补解释能力
- 结果解释
- 场景影响解释
- 决策建议

## 不建议现在做的事情

当前阶段不建议：
- 大规模重写前端结构
- 重新设计一整套路由体系
- 引入复杂数据库来替代 CSV
- 立刻把所有技术问题都当成必须重构的问题

原因很简单：
- 当前系统已经能承担“展示平台”的任务
- 未来最关键的不是技术重构，而是业务表达升级

## 一句话总结

这个项目现在已经完成了“算法展示平台”的主骨架，后续最值得投入的方向不是重做代码，而是把现有页面逐步补充成更懂业务、更能解释结果、更能支持决策的展示系统。
